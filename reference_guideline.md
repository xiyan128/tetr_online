# Final Implementation Spec: Guideline-Compatible Falling-Block Puzzle Engine

## 0. What this spec is

This is a self-contained implementation contract for building a modern guideline-style falling-block puzzle game from scratch, based on the supplied 2009 design guideline and the alternate architecture draft.

### 0.1 Branding boundary

The source design document contains official Tetris branding, legal-screen, logo, and music guidance. Do not ship official Tetris trademarks, logos, trade dress, legal copy, or licensed music unless the project is authorized to use them.

For an unaffiliated implementation:

* Implement the mechanics described here.
* Use original branding, original assets, and original music.
* Keep all code/entity names generic when useful, such as `Matrix`, `Tetrimino`, `Mino`, `LineClear`, etc., but do not present the product as official Tetris.

### 0.2 Non-negotiable engine invariants

* The visible Matrix is exactly 10 columns by 20 rows.
* There is a hidden 10 by 20 Buffer Zone above the Matrix.
* Only one active Tetrimino is player-controlled at a time.
* Pieces are generated by a seven-bag randomizer.
* Pieces spawn North Facing above the Skyline.
* Generated pieces immediately drop one row if not blocked below.
* SRS rotation uses five ordered tests and the first non-colliding result wins.
* Default Lock Down mode is Extended Placement.
* Default Hold Queue and Ghost Piece are on.
* Hard Drop locks immediately.
* Soft Drop is 20x normal fall speed.
* Standard line clears are detected only inside the visible Matrix rows 1 through 20.
* Game over is caused by Block Out, Lock Out, or Top Out.

# 1. Coordinate system and board model

## 1.1 Coordinate system

Use a Cartesian grid:

* `+X` points right.
* `+Y` points up.
* The bottom-left visible cell is `(1, 1)`.
* The upper-right visible cell is `(10, 20)`.
* Row `20` is the top visible Matrix row.
* Row `21` is the first hidden Buffer Zone row.

## 1.2 Matrix, Skyline, Buffer Zone, Top Out Line

| Region       | X range |  Y range | Visible? | Purpose                            |
| ------------ | ------: | -------: | -------- | ---------------------------------- |
| Matrix       |   1..10 |    1..20 | yes      | Normal gameplay and line clears    |
| Buffer Zone  |   1..10 |   21..40 | no       | Spawn area and game-over detection |
| Top Out Line |     n/a | above 40 | no       | Absolute ceiling for Top Out       |

Definitions:

* **Matrix:** visible active play area, 10 wide by 20 high.
* **Skyline:** boundary between row 20 and row 21.
* **Buffer Zone:** invisible 10 by 20 area above the Skyline.
* **Top Out Line:** ceiling above row 40.
* **Mino:** one square unit.
* **Tetrimino:** shape made of exactly four Minos.
* **Block:** a Mino that has locked into the board.
* **Surface:** the Matrix floor or the top of locked Blocks.

## 1.3 Internal board representation

Represent the logical board as 10 by 40, not 10 by 20.

```ts
type PieceType = 'I' | 'J' | 'L' | 'O' | 'S' | 'T' | 'Z';
type Facing = 0 | 1 | 2 | 3; // 0=N, 1=E, 2=S, 3=W

type Cell = {
  occupied: boolean;
  type?: PieceType;
  solidGarbage?: boolean;
};

type Board = Cell[][]; // rows 1..40, cols 1..10 via helper accessors
```

Recommended helper functions:

```ts
function isInsideX(x: number): boolean { return x >= 1 && x <= 10; }
function isVisibleY(y: number): boolean { return y >= 1 && y <= 20; }
function isBufferY(y: number): boolean { return y >= 21 && y <= 40; }
function isInsideLogicalY(y: number): boolean { return y >= 1 && y <= 40; }
function isAboveTopOutLine(y: number): boolean { return y > 40; }
```

Collision policy:

* `x < 1` or `x > 10` collides with a wall.
* `y < 1` collides with the floor.
* Occupied cells collide.
* Active pieces should not lock above `y=40`.
* Existing Blocks pushed above `y=40` cause Top Out.

# 2. Tetriminos, colors, facings, and shape data

## 2.1 Tetrimino set

| Type | Color      | North-facing shape description                        |
| ---- | ---------- | ----------------------------------------------------- |
| I    | light blue | four Minos in a horizontal line                       |
| J    | dark blue  | row of three with one above the left side             |
| L    | orange     | row of three with one above the right side            |
| O    | yellow     | 2x2 square                                            |
| S    | green      | two stacked horizontal dominoes, top one offset right |
| T    | purple     | row of three with one above center                    |
| Z    | red        | two stacked horizontal dominoes, top one offset left  |

On monochrome or limited-color hardware, pieces must still be visually distinguishable by hue, pattern, or texture.

## 2.2 Facing constants

```ts
const NORTH = 0;
const EAST = 1;
const SOUTH = 2;
const WEST = 3;
```

Rotation direction:

```ts
function rotateFacing(facing: Facing, dir: 'CW' | 'CCW'): Facing {
  return dir === 'CW'
    ? ((facing + 1) % 4) as Facing
    : ((facing + 3) % 4) as Facing;
}
```

## 2.3 Spawn coordinates

All pieces spawn North Facing.

Use these exact initial occupied cells before the immediate-drop rule:

| Piece | Spawn cells                      |
| ----- | -------------------------------- |
| I     | `(4,21), (5,21), (6,21), (7,21)` |
| J     | `(4,22), (4,21), (5,21), (6,21)` |
| L     | `(4,21), (5,21), (6,21), (6,22)` |
| O     | `(5,21), (6,21), (5,22), (6,22)` |
| S     | `(4,21), (5,21), (5,22), (6,22)` |
| T     | `(4,21), (5,21), (6,21), (5,22)` |
| Z     | `(4,22), (5,22), (5,21), (6,21)` |

Spawn summary:

* J, L, S, T, Z span columns 4 through 6 and rows 21 and 22.
* I spans columns 4 through 7 on row 21 only.
* O spans columns 5 and 6 on rows 21 and 22.

## 2.4 Active-piece representation

```ts
type ActivePiece = {
  type: PieceType;
  facing: Facing;
  origin: { x: number; y: number };

  lockTimer: number;
  lockTimerActive: boolean;
  landed: boolean;
  lowestYReached: number;
  groundedMoveRotateCountSinceLowest: number;

  holdUsedOnThisPiece: boolean;

  lastSuccessfulAction:
    | 'spawn'
    | 'fall'
    | 'move'
    | 'rotate'
    | 'softDrop'
    | 'hardDrop'
    | 'holdSwap';

  lastRotationDirection?: 'CW' | 'CCW';
  lastRotationKickIndex?: 1 | 2 | 3 | 4 | 5;

  // Once a T-piece uses SRS test 5 to enter a T-slot, later rotations before lock
  // should not downgrade it to Mini under the guideline's point-5 exception.
  usedKick5IntoTSlot: boolean;
};
```

Use `origin` as the piece's rotation reference. The concrete cell offsets below must be consistent with the spawn cells above.

The following representation is easy to implement with the SRS tables in this spec.

North-facing cells relative to `origin`:

```ts
const NORTH_CELLS = {
```

| Piece         |   Origin |
| ------------- | -------: |
| T, J, L, S, Z | `(5,21)` |
| O             | `(5,21)` |

* CW: `(x, y) -> (y, -x)`

```ts
const I_CELLS = {
```

For O, the footprint remains identical on all facings.

# 3. Random generation and queues

## 3.1 Seven-bag randomizer

Use a seven-bag system:

1. Create a bag containing exactly `I, J, L, O, S, T, Z`.
2. Shuffle uniformly with the game RNG.
3. Deal pieces from the bag into the Next Queue.
4. When the bag is empty, create and shuffle a new bag.

Use a seeded RNG in tests and multiplayer.

## 3.2 Next Queue

* Default visible count: 6 if screen space allows.
* Legal visible range: 1 through 6.
* Tetriminos in the queue are displayed North Facing.
* The next Tetrimino to enter the Matrix is the one closest to the top of the Matrix if the queue is vertical.
* Queue animation is allowed only if the pieces remain easy to distinguish in peripheral vision.

Initialize the queue with at least `visibleNextCount + 1` pieces so the queue can remain filled immediately after generation.

## 3.3 Hold Queue

* Default: enabled.
* Holds at most one Tetrimino.
* Held Tetrimino is displayed North Facing.
* When swapped out, the held Tetrimino respawns at its normal generation position and North Facing.
* A Lock Down must occur between Holds. The player cannot Hold twice on the same active Tetrimino.

# 4. Generation and spawn flow

## 4.1 Generation ordering

When generating a new active Tetrimino:

1. Wait for the configured generation delay, default `0.2s`, after the previous Completion Phase.
2. Pop the front of the Next Queue.
3. Append a new piece from the bag to the back of the Next Queue.
4. Place the new active piece at its spawn cells, North Facing.
5. Check Block Out at the spawn cells.
6. If no Block Out, apply the immediate-drop rule.
7. Enable player control.
8. Compute Ghost Piece if enabled.

Important correction: **Block Out is checked against the generation cells before the immediate one-row drop.** Do not let the immediate drop hide an invalid spawn overlap.

## 4.2 Immediate-drop rule

After successful spawn:

* If the piece can move down by exactly one row without colliding, immediately move it down one row.
* If an existing Block is in its downward path, it does not drop immediately.
* If it cannot drop because of Blocks below, the renderer may show only a few pixels above the Skyline, but the logical piece remains at generation position.

# 5. Core engine state machine

## 5.1 Phases

Each active Tetrimino moves through these phases:

1. Generation
2. Falling
3. Lock
4. Pattern
5. Iterate
6. Animate
7. Eliminate
8. Completion

Only one Tetrimino can be manipulated by the player at a time.

## 5.2 Generation Phase

Responsibilities:

* Generate from Next Queue / seven-bag.
* Spawn North Facing.
* Check Block Out.
* Apply immediate one-row drop.
* Initialize lock state.
* Initialize per-piece Hold permission.
* Initialize T-Spin tracking.
* Show Ghost Piece.

## 5.3 Falling Phase

Process:

* Apply gravity according to current fall speed.
* Process left/right movement, rotation, Soft Drop, Hard Drop, and Hold.
* If Hard Drop occurs, lock immediately and continue to post-lock processing.
* If the piece contacts a Surface, enter Lock Phase.

## 5.4 Lock Phase

When the active Tetrimino lands on a Surface:

* Start a `0.5s` Lock Down Timer.
* Continue accepting movement, rotation, Soft Drop, Hard Drop, and Hold if legal.
* If the piece moves off the Surface and has space to fall, return to Falling Phase.
* If timer expires, Lock Down.
* Timer reset behavior depends on the selected lock-down mode.

## 5.5 Pattern Phase

After Lock Down:

* Scan rows `1..20` only for complete horizontal rows.
* A row is complete when all 10 visible Matrix cells are occupied.
* Mark complete rows for removal.

Do not scan rows `21..40` for normal Line Clears. The Buffer Zone exists for spawn and game-over detection, not normal line-clearing.

## 5.6 Iterate Phase

Variant hook. Core implementation can treat it as no-op.

## 5.7 Animate Phase

Play line-clear and action animations. Logical state may wait until animation completes, but scoring/action classification should already be determined or cached.

## 5.8 Eliminate Phase

* Remove all marked rows.
* Collapse all Blocks above cleared rows down by the number of cleared rows below them.
* Collapse only vertical positions. Do not reorder columns.

## 5.9 Completion Phase

* Update score, level, goal, time, stats, high score candidate state, and UI.
* Resolve variant completion.
* Schedule the next Generation Phase after `0.2s`, unless game over.

## 5.10 Recommended post-lock ordering

On Lock Down:

1. Convert active Minos to locked Blocks.
2. Classify possible T-Spin/Mini T-Spin using cached last-rotation state and final board context.
3. Check Lock Out.
4. Detect visible line clears.
5. Score the action and update B2B.
6. Update variable-goal awarded units.
7. In multiplayer, compute outgoing attack, resolve Receiving Queue counterattack, then insert remaining incoming lines.
8. Check Top Out after garbage insertion.
9. Animate and eliminate visible line clears.
10. Collapse visible Matrix Blocks.
11. Update level/variant completion.
12. Clear per-piece Hold restriction for the next piece.

# 6. Movement, DAS, rotation input, drops, and Hold

## 6.1 Grid movement

* Left/right movement changes x by exactly 1 cell.
* Gravity, Soft Drop, and Hard Drop move downward.
* No normal upward translation exists.
* SRS may move a piece upward as part of rotation.
* Illegal movement into walls, floor, or occupied cells fails and leaves the piece unchanged.
* Rendered motion may be smooth, but logical positions are cell-snapped.

## 6.2 DAS / Auto-repeat

Left/right movement supports delayed auto-shift:

* Tap: one-cell movement.
* Hold delay: about `0.3s` before auto-repeat begins.
* Repeat speed: fast enough to move a piece from one side of the Matrix to the other in about `0.5s` after DAS begins.
* Auto-repeat applies only to left/right movement.
* Auto-repeat carries into the next Tetrimino if the direction button remains held after Lock Down.
* If the opposite direction is pressed while one direction is held, start the new direction with the initial `0.3s` delay.
* If both directions are held and one is released, move in the remaining direction after reapplying the initial delay.

Recommended repeat interval: 50 to 60 ms per cell, tunable to hit the side-to-side feel target.

## 6.3 Rotation input

* Each rotation button press attempts exactly one 90-degree rotation.
* No rotation auto-repeat.
* Rotation can happen while left/right auto-repeat is active.
* Use SRS five-test order.
* If all tests fail, play optional failure feedback and leave the piece unchanged.

## 6.4 Hard Drop

Hard Drop:

* Moves the active piece straight down to the first Surface below it.
* Locks immediately.
* Logical duration: `0.0001s`.
* No auto-repeat.
* Awards `2 * m` points, where `m` is the number of rows descended due to Hard Drop.
* Should optionally render a hard-drop trail.

## 6.5 Soft Drop

Soft Drop:

* While held, fall speed becomes `normalFallSpeed / 20` seconds per line.
* Releasing Soft Drop restores normal fall speed.
* If Soft Drop lands on a Surface, it does **not** lock immediately. It uses the normal Lock Down Timer.
* Soft Drop continues into the next Tetrimino if the button remains held.
* Awards `1 * n` points, where `n` is the number of rows descended by player-controlled Soft Drop.

## 6.6 Hold behavior

When Hold is pressed:

1. If Hold is disabled, ignore.
2. If `holdUsedOnThisPiece` is true, ignore.
3. If Hold Queue is empty:

   * Store active piece type in Hold Queue.
   * Generate the next piece from the Next Queue.
4. If Hold Queue contains a piece:

   * Swap active piece type with held type.
   * Respawn swapped-in piece at its normal generation position, North Facing.
5. Set `holdUsedOnThisPiece = true` for the new active piece instance.
6. Increment Hold stats.

Hold does not break a Back-to-Back sequence.

# 7. Super Rotation System (SRS)

## 7.1 SRS behavior

On a rotation input:

1. Determine target facing.
2. Try up to five tests in table order.
3. Each test applies a candidate offset `(dx, dy)` to the rotated piece.
4. The first candidate with no collision succeeds.
5. If all five collide, rotation fails.
6. Record successful test index `1..5` for T-Spin recognition.

Coordinate convention:

* `+X` is right.
* `+Y` is up.
* Offsets apply to the piece origin after rotating to the target facing.

## 7.2 SRS table for J, L, S, T, Z

| Rotation  |  Test 1 |   Test 2 |    Test 3 |   Test 4 |    Test 5 |
| --------- | ------: | -------: | --------: | -------: | --------: |
| 0->1 N->E | `(0,0)` | `(-1,0)` | `(-1,+1)` | `(0,-2)` | `(-1,-2)` |
| 1->0 E->N | `(0,0)` | `(+1,0)` | `(+1,-1)` | `(0,+2)` | `(+1,+2)` |
| 1->2 E->S | `(0,0)` | `(+1,0)` | `(+1,-1)` | `(0,+2)` | `(+1,+2)` |
| 2->1 S->E | `(0,0)` | `(-1,0)` | `(-1,+1)` | `(0,-2)` | `(-1,-2)` |
| 2->3 S->W | `(0,0)` | `(+1,0)` | `(+1,+1)` | `(0,-2)` | `(+1,-2)` |
| 3->2 W->S | `(0,0)` | `(-1,0)` | `(-1,-1)` | `(0,+2)` | `(-1,+2)` |
| 3->0 W->N | `(0,0)` | `(-1,0)` | `(-1,-1)` | `(0,+2)` | `(-1,+2)` |
| 0->3 N->W | `(0,0)` | `(+1,0)` | `(+1,+1)` | `(0,-2)` | `(+1,-2)` |

## 7.3 SRS table for I

| Rotation  |  Test 1 |   Test 2 |   Test 3 |    Test 4 |    Test 5 |
| --------- | ------: | -------: | -------: | --------: | --------: |
| 0->1 N->E | `(0,0)` | `(-2,0)` | `(+1,0)` | `(-2,-1)` | `(+1,+2)` |
| 1->0 E->N | `(0,0)` | `(+2,0)` | `(-1,0)` | `(+2,+1)` | `(-1,-2)` |
| 1->2 E->S | `(0,0)` | `(-1,0)` | `(+2,0)` | `(-1,+2)` | `(+2,-1)` |
| 2->1 S->E | `(0,0)` | `(+1,0)` | `(-2,0)` | `(+1,-2)` | `(-2,+1)` |
| 2->3 S->W | `(0,0)` | `(+2,0)` | `(-1,0)` | `(+2,+1)` | `(-1,-2)` |
| 3->2 W->S | `(0,0)` | `(-2,0)` | `(+1,0)` | `(-2,-1)` | `(+1,+2)` |
| 3->0 W->N | `(0,0)` | `(+1,0)` | `(-2,0)` | `(+1,-2)` | `(-2,+1)` |
| 0->3 N->W | `(0,0)` | `(-1,0)` | `(+2,0)` | `(-1,+2)` | `(+2,-1)` |

## 7.4 O rotation

The O piece keeps the same footprint. A rotation may update facing state internally, but the occupied cells should remain unchanged. No kicks are needed.

## 7.5 Rotation pseudocode

```ts
function tryRotate(piece: ActivePiece, dir: 'CW' | 'CCW', board: Board): boolean {
  const from = piece.facing;
  const to = rotateFacing(from, dir);
  const kicks = getKickOffsets(piece.type, from, to);

  for (let i = 0; i < kicks.length; i++) {
    const [dx, dy] = kicks[i];
    const candidate = clone(piece);
    candidate.facing = to;
    candidate.origin.x += dx;
    candidate.origin.y += dy;

    if (!collides(candidate, board)) {
      piece.facing = to;
      piece.origin = candidate.origin;
      piece.lastSuccessfulAction = 'rotate';
      piece.lastRotationDirection = dir;
      piece.lastRotationKickIndex = (i + 1) as 1 | 2 | 3 | 4 | 5;

      if (piece.type === 'T' && i + 1 === 5 && isTSlot(piece, board)) {
        piece.usedKick5IntoTSlot = true;
      }

      onSuccessfulGroundedMoveOrRotate(piece);
      return true;
    }
  }

  return false;
}
```

# 8. Lock Down modes

## 8.1 Shared Lock Down rules

* Hard Drop locks immediately.
* Natural fall or Soft Drop onto a Surface starts a `0.5s` Lock Down Timer.
* If the piece is no longer touching a Surface, timer is not actively locking it and the game returns to Falling Phase.
* Default mode is Extended Placement.
* Multiplayer and arcade-style variants must use Extended Placement.

## 8.2 Extended Placement Lock Down

Default.

Rules:

1. When the piece lands, start timer at `0.5s`.
2. Each successful grounded left/right movement or rotation resets the timer to `0.5s`, up to the movement budget.
3. The piece gets at most 15 successful grounded left/right movements or rotations after landing.
4. If the piece falls one row below the lowest row it has ever reached, reset the 15-action counter.
5. If the 15-action counter is exhausted, further grounded movement/rotation must not reset the timer.
6. If the piece is airborne after exhausting the counter, it may still move/rotate, but once it touches a Surface it locks without further extension unless it has fallen below the previous lowest row.

Implementation state:

```ts
lowestYReached: number;
groundedMoveRotateCountSinceLowest: number; // max 15 before no more resets
```

Use the lowest occupied Mino y-coordinate or origin y consistently. Tests must use the same convention.

## 8.3 Infinite Placement Lock Down

Rules:

1. When the piece lands, start timer at `0.5s`.
2. Any successful movement or rotation that actually changes position or orientation resets timer to `0.5s`.
3. There is no 15-action limit.

## 8.4 Classic Lock Down

Rules:

1. When the piece lands, start timer at `0.5s`.
2. Timer resets only if the piece falls farther down in the Matrix.
3. Horizontal movement and rotation do not reset the timer by themselves.

# 9. Gravity, levels, and drop speeds

## 9.1 Normal fall speed formula

Normal Fall Speed is seconds per line:

```txt
fallSpeed(level) = (0.8 - ((level - 1) * 0.007)) ^ (level - 1)
```

Approximate values:

| Level | sec/line |
| ----: | -------: |
|     1 |    1.000 |
|     2 |    0.793 |
|     3 |    0.618 |
|     4 |    0.473 |
|     5 |    0.355 |
|     6 |    0.262 |
|     7 |    0.190 |
|     8 |    0.135 |
|     9 |    0.094 |
|    10 |    0.064 |
|    11 |    0.043 |
|    12 |    0.028 |
|    13 |    0.018 |
|    14 |    0.011 |
|    15 |    0.007 |

## 9.2 Soft and Hard Drop speeds

```txt
softDropSpeed(level) = fallSpeed(level) / 20
hardDropDuration = 0.0001s
```

## 9.3 Gravity update pseudocode

```ts
function updateGravity(state: GameState, dt: number) {
  const piece = state.active;
  const speed = state.input.softDropHeld
    ? fallSpeed(state.level) / 20
    : fallSpeed(state.level);

  state.gravityAccumulator += dt;

  while (state.gravityAccumulator >= speed) {
    if (canMove(piece, 0, -1, state.board)) {
      piece.origin.y -= 1;
      piece.lastSuccessfulAction = state.input.softDropHeld ? 'softDrop' : 'fall';
      updateLowestYAndLockBudget(piece);
      if (state.input.softDropHeld) state.score += 1;
      state.gravityAccumulator -= speed;
    } else {
      enterOrContinueLockPhase(state);
      break;
    }
  }
}
```

At high levels, multiple rows may fall within one frame. Always process row-by-row to preserve collision correctness.

# 10. Ghost Piece

The Ghost Piece shows where the active Tetrimino would land if Hard Dropped from its current position.

Requirements:

* Default: on.
* Optionally disabled by player.
* May render as outline or translucent copy.
* Recomputed after spawn, movement, rotation, gravity, Soft Drop, Hold swap, and garbage insertion.
* Must not obscure active piece or locked Blocks.

Algorithm:

```ts
function computeGhost(piece: ActivePiece, board: Board): ActivePiece {
  const ghost = clone(piece);
  while (canMove(ghost, 0, -1, board)) {
    ghost.origin.y -= 1;
  }
  return ghost;
}
```

# 11. Line clears and collapse

## 11.1 Detection

After Lock Down, scan only visible Matrix rows `1..20`.

A row clears if all 10 cells are occupied.

## 11.2 Action names by cleared visible lines

| Cleared rows | Action                                            |
| -----------: | ------------------------------------------------- |
|            0 | no Line Clear, but T-Spin actions may still score |
|            1 | Single                                            |
|            2 | Double                                            |
|            3 | Triple                                            |
|            4 | Tetris                                            |

## 11.3 Collapse

If rows are cleared:

1. Remove all cells in cleared rows.
2. For every remaining Block above a cleared row, move it downward by the number of cleared rows below it.
3. Preserve column positions.
4. Do not create gaps except those already in non-cleared rows.

# 12. T-Spin and Mini T-Spin recognition

## 12.1 T-Slot definition

A T-Slot exists when, in the final T-piece position, at least three of the four diagonal cells around the T center are blocked.

Diagonal corners relative to T center/origin:

```ts
NW = (-1, +1)
NE = (+1, +1)
SW = (-1, -1)
SE = (+1, -1)
```

A corner is blocked if:

* It contains a locked Block.
* It is outside the horizontal Matrix walls.
* It is below the floor.

The top of the Buffer Zone is not normally needed for T-Spin recognition, but out-of-logical-board cells should be treated as blocked if encountered.

## 12.2 Rotation precondition

A T-Spin or Mini T-Spin requires:

* The locked piece is a T-Tetrimino.
* The player successfully rotated the T-Tetrimino before Lock Down.
* The final position is a T-Slot.

Implementation choice for determinism:

* Use `lastSuccessfulAction === 'rotate'` as the primary recognition rule.
* Also preserve the guideline-specific Point-5 exception via `usedKick5IntoTSlot`.

This prevents accidental false positives where the player rotates, then slides or drops into a slot without the final placement being caused by rotation.

## 12.3 Corner labels by facing

Treat A/B as the front corners and C/D as the back corners for the final facing.

| Facing | A  | B  | C  | D  |
| ------ | -- | -- | -- | -- |
| North  | NW | NE | SW | SE |
| East   | NE | SE | NW | SW |
| South  | SE | SW | NE | NW |
| West   | SW | NW | SE | NE |

## 12.4 Full T-Spin

Classify as full T-Spin if the rotation precondition and T-Slot condition are true, and any of these are true:

1. A and B are blocked, and at least one of C or D is blocked.
2. The T fully fills a T-Slot with no holes.
3. SRS Test 5 was used to rotate into the T-Slot, or `usedKick5IntoTSlot` was already set for this active piece.

Point-5 override is critical: if Test 5 put the T into the T-Slot, award full T-Spin, not Mini.

## 12.5 Mini T-Spin

Classify as Mini T-Spin if full T-Spin did not apply, and either:

1. C and D are blocked, and at least one of A or B is blocked.
2. The T creates holes in a T-Slot.

## 12.6 T-Spin action table

| Classification | Lines cleared | Action             |
| -------------- | ------------: | ------------------ |
| Mini           |             0 | Mini T-Spin        |
| Mini           |             1 | Mini T-Spin Single |
| Full           |             0 | T-Spin             |
| Full           |             1 | T-Spin Single      |
| Full           |             2 | T-Spin Double      |
| Full           |             3 | T-Spin Triple      |

A T-Spin clearing four lines is not defined by the guideline and should be treated as invalid/impossible under normal geometry. If it occurs from a bug or custom variant, log it explicitly.

## 12.7 T-Spin pseudocode

```ts
function classifyTSpin(piece: ActivePiece, board: Board, linesCleared: number): TSpinKind {
  if (piece.type !== 'T') return 'NONE';
  if (piece.lastSuccessfulAction !== 'rotate' && !piece.usedKick5IntoTSlot) return 'NONE';

  const corners = getTCornerBlockedMap(piece, board); // A,B,C,D booleans
  const slot = countTrue(corners.A, corners.B, corners.C, corners.D) >= 3;
  if (!slot) return 'NONE';

  const fullByCorners = corners.A && corners.B && (corners.C || corners.D);
  const fullByKick5 = piece.usedKick5IntoTSlot || piece.lastRotationKickIndex === 5;
  const fullByNoHoleSlot = fillsTSlotWithoutHoles(piece, board);

  if (fullByKick5 || fullByCorners || fullByNoHoleSlot) return 'FULL';

  const miniByCorners = corners.C && corners.D && (corners.A || corners.B);
  const miniByHoles = createsHolesInTSlot(piece, board);

  if (miniByCorners || miniByHoles) return 'MINI';
  return 'NONE';
}
```

# 13. Scoring and Back-to-Back

## 13.1 Base scoring table

Line clears, T-Spins, and Mini T-Spins are multiplied by current Level. Drop scores are not.

| Action             |                 Points |
| ------------------ | ---------------------: |
| Single             |          `100 * Level` |
| Double             |          `300 * Level` |
| Triple             |          `500 * Level` |
| Tetris             |          `800 * Level` |
| Mini T-Spin        |          `100 * Level` |
| Mini T-Spin Single |          `200 * Level` |
| T-Spin             |          `400 * Level` |
| T-Spin Single      |          `800 * Level` |
| T-Spin Double      |         `1200 * Level` |
| T-Spin Triple      |         `1600 * Level` |
| Soft Drop          | `1 * cellsSoftDropped` |
| Hard Drop          | `2 * cellsHardDropped` |

## 13.2 Score Back-to-Back rules

Qualifying score B2B actions:

* Tetris.
* Mini T-Spin Single.
* T-Spin Single.
* T-Spin Double.
* T-Spin Triple.

Rules:

* First qualifying action starts the B2B chain and gets no B2B bonus.
* Each subsequent qualifying action while B2B is active gets `0.5 * actionBaseScore` bonus.
* Single, Double, and Triple break the B2B chain.
* Zero-line T-Spins and zero-line Mini T-Spins do not receive B2B bonus.
* Zero-line T-Spins and zero-line Mini T-Spins cannot start B2B.
* Zero-line T-Spins, zero-line locks, and Hold do not break an existing B2B chain.

## 13.3 B2B example

At Level 1:

```txt
Tetris              = 800, starts B2B, no bonus
T-Spin Double       = 1200 + 600 B2B
T-Spin              = 400, no B2B, chain preserved
Tetris              = 800 + 400 B2B
T-Spin Single       = 800 + 400 B2B
Total               = 5400
```

## 13.4 Scoring pseudocode

```ts
function applyScoreForAction(state: GameState, action: Action, dropPoints: number) {
  const base = baseScore(action, state.level);
  let bonus = 0;

  const qualifies = isB2BQualifyingLineClear(action);
  const breaks = action === 'Single' || action === 'Double' || action === 'Triple';

  if (qualifies) {
    if (state.backToBackActive) {
      bonus = Math.floor(base * 0.5);
      state.stats.backToBackAwards += 1;
    } else {
      state.backToBackActive = true;
    }
  } else if (breaks) {
    state.backToBackActive = false;
  }

  state.score += base + bonus + dropPoints;
}
```

# 14. Levels and goals

## 14.1 Level range

Typical levels are 1 through 15.

Level affects:

* Normal Fall Speed.
* Score multiplier.

Default starting level: 1.

## 14.2 Fixed Goal System

Each level normally requires 10 physical Line Clears.

* Total to clear levels 1 through 15: 150 lines.
* If starting at Level `L > 1`, first goal is `10 * L` lines, accounting for prior skipped levels.
* After that first level-up, goal returns to 10 lines per level.

Example:

* Start Level 4.
* Need 40 lines to reach Level 5.
* After reaching Level 5, need 10 lines for next level.

## 14.3 Variable Goal System

Goal for a level:

```txt
Goal = Level * 5
```

This uses awarded line-clear units, not physical lines.

Playing Level 1 through 15 requires total awarded units:

```txt
5 * (1 + 2 + ... + 15) = 600
```

If starting at Level 2 or higher, the first goal is simply `Level * 5`.

| Action             |       Awarded units |

| Single             |                   1 |
| Mini T-Spin        |                   1 |
| Mini T-Spin Single |                   2 |
| Double             |                   3 |
| T-Spin             |                   4 |
| Triple             |                   5 |
| Tetris             |                   8 |
| T-Spin Single      |                   8 |
| T-Spin Double      |                  12 |

# 16. Game-over conditions

## 16.1 Block Out

Block Out occurs during Generation Phase when any spawn cell of the newly generated Tetrimino overlaps an existing Block.

## 16.2 Lock Out

Lock Out occurs when the active Tetrimino Locks Down completely above the Skyline.

Implementation:

```ts
const allMinosAboveSkyline = lockedCells.every(c => c.y >= 21);
if (allMinosAboveSkyline) gameOver('LockOut');
```

## 16.3 Top Out

Top Out occurs when incoming lines, usually multiplayer garbage, force existing Blocks above the Top Out Line, i.e. above row 40.

## 16.4 Not game over

These cases are allowed:

* A piece locks entirely inside visible Matrix rows 1 through 20.
* A piece locks partly below and partly above the Skyline.
* The active piece is manipulated above the Skyline before Lock Down.
* Existing Blocks are above the Skyline but still within the Buffer Zone.

## 16.5 Recommended check order

1. On generation: Block Out.
2. On Lock Down: Lock Out.
3. On garbage insertion: Top Out.
4. Otherwise continue.

# 22. Multiplayer Line Attack and Counter Attack

## 22.1 Line types

Broken Lines:

* Clearable.
* Usually contain one empty gap cell.

Solid Lines:

* No gaps.
* Typically unclearable.

Typical multiplayer uses Broken Lines.

## 22.2 Receiving Queue

Each player has a Receiving Queue:

* Visual bar/indicator next to the Matrix.
* Stores incoming lines before insertion.
* Incoming lines appear from bottom upward only after the player’s Tetrimino Locks Down.

## 22.3 Broken-line gap randomness

* After every eight new Broken Lines appear in a player’s Matrix, choose a different random gap column for the next group of eight.
* Queued incoming lines do not count until they actually appear in the Matrix.
* When lines are sent to multiple opponents, their line patterns are identical.

Implementation:

```ts
function chooseGarbageGap(player: PlayerState): number {
  if (player.garbageLinesAppeared % 8 === 0) {
    let next;
    do next = randomInt(1, 10); while (next === player.currentGarbageGap);
    player.currentGarbageGap = next;
  }
  return player.currentGarbageGap;
}
```

## 22.4 Attack table

| Action             | Lines sent |
| ------------------ | ---------: |
| Single             |          0 |
| Double             |          1 |
| Triple             |          2 |
| Tetris             |          4 |
| T-Spin             |          0 |
| Mini T-Spin        |          0 |
| Mini T-Spin Single |          0 |
| T-Spin Single      |          2 |
| T-Spin Double      |          4 |
| T-Spin Triple      |          6 |
| Back-to-Back       |         +1 |

Multiplayer B2B attack bonus is `+1` line. Keep this separate from scoring B2B.

## 22.5 Counter Attack resolution

If the player has incoming lines and performs an attack:

```ts
net = outgoingAttackLines - receivingQueueLines;

if (net > 0) {
  receivingQueueLines = 0;
  sendLinesToTargets(net);
} else if (net < 0) {
  receivingQueueLines = -net;
  insertIncomingLinesAfterLockDown(receivingQueueLines);
  receivingQueueLines = 0;
} else {
  receivingQueueLines = 0;
}
```

Examples:

* Receiving 3, Tetris sends 4 -> send 1, receive 0.
* Receiving 5, Tetris sends 4 -> send 0, insert 1.
* Receiving 2, T-Spin Triple sends 6 -> send 4, receive 0.
* Receiving 4, Single sends 0 -> insert 4.

## 22.6 Garbage insertion

When inserting incoming lines:

1. Push all existing Blocks upward by the number of inserted lines.
2. Insert new lines from bottom upward.
3. Use chosen gap for Broken Lines.
4. If any existing Block is pushed above row 40, Top Out.
5. Recompute Ghost Piece after insertion.

## 24.3 Update loop

```ts
function update(state: GameState, input: InputFrame, dt: number) {
  if (state.phase === 'paused') return;

  updateScreenStateIfNotPlaying(state, input);
  if (state.phase !== 'playing') return;

  updateClocks(state, dt);
  if (checkTimedVariantEnd(state)) return;

  switch (state.enginePhase) {
    case 'generation':
      updateGeneration(state, dt);
      break;
    case 'falling':
      processInputs(state, input, dt);
      updateGravity(state, dt);
      updateGhost(state);
      break;
    case 'lock':
      processInputs(state, input, dt);
      updateGravityIfSoftDropOrFallPossible(state, dt);
      updateLockTimer(state, dt);
      updateGhost(state);
      break;
    case 'pattern':
      detectLineClears(state);
      state.enginePhase = 'iterate';
      break;
    case 'iterate':
      runVariantIterateHook(state);
      state.enginePhase = 'animate';
      break;
    case 'animate':
      updateAnimationsOrAdvance(state, dt);
      break;
    case 'eliminate':
      eliminateAndCollapse(state);
      state.enginePhase = 'completion';
      break;
    case 'completion':
      completePieceCycle(state);
      break;
  }
}
```

## 25.9 Goals

Fixed:

* Level 1 goal 10.
* Starting at Level 4 first goal 40.
* Later fixed goals are 10.

Variable:

* Goal is `Level * 5`.
* Level 1 goal 5.
* Level 15 goal 75.
* Total from levels 1..15 is 600.
* Tetris and T-Spin Single award 8 units.
* T-Spin Double awards 12.
* T-Spin Triple awards 16.

## 25.10 Game over

* Spawn overlap -> Block Out.
* Whole piece locked above Skyline -> Lock Out.
* Piece partly below Skyline -> not Lock Out.
* Blocks in Buffer Zone below row 41 -> not Top Out.
* Garbage pushing Blocks above row 40 -> Top Out.

## 25.11 Variants

Marathon:

* Levels advance by selected goal system.
* Ends on game over or configured Level 15 completion.

Sprint:

* No Level Up.
* Ends at target physical line count, default 40.

Ultra:

* No Level Up.
* Ends when timer expires.

## 25.12 Multiplayer

* Double sends 1.
* Triple sends 2.
* Tetris sends 4.
* T-Spin Single sends 2.
* T-Spin Double sends 4.
* T-Spin Triple sends 6.
* Mini T-Spin Single sends 0.
* B2B adds +1 attack line.
* Incoming queue is reduced before outgoing lines are sent.
* Remaining incoming lines insert after Lock Down.
* Broken-line gap changes after every eight appeared lines, not queued lines.
* Multi-target attacks send identical line patterns to all recipients.

# 26. Deliberate implementation choices where the source is underspecified

Use these defaults unless product direction overrides them:

1. **Line-clear scan range:** rows 1..20 only. Buffer rows do not clear in core modes.
2. **T-Spin last-action rule:** require final successful action to be rotation, while preserving Point-5 override. This is deterministic and avoids false positives.
3. **Ultra time:** configurable. Default to 2 minutes if no product setting is given.
4. **Sprint target:** 40 physical lines.
5. **High score table size:** top 10 per variant.
6. **Combo scoring:** none by default, because not specified.
7. **Line-clear animation duration:** platform-tuned, but logical elimination waits for animation completion if animations are enabled.
8. **Starting Blocks:** optional variant hook, disabled in standard Marathon/Sprint/Ultra unless configured.
9. **Official branding/music/legal copy:** hooks only unless licensed.

# 27. Build checklist

A build is spec-complete when all of the following are true:

Current audit notes are based on the Rust/Bevy implementation under `src/`.

* [x] 10x20 visible Matrix plus 10x20 Buffer Zone.
  * Current: `Board::with_top_margin(config.board_width, config.board_height, 20)` allocates a 10x40 logical board, while visible rendering and row clearing use `board_height = 20`.
  * Gap: `Board::get_cell_kind` should treat `y < 0` as floor collision. It currently treats negative Y as empty, which can let gravity, Ghost Piece, and Hard Drop move below the Matrix.
* [ ] Exact spawn cells and immediate-drop rule.
  * Current: pieces spawn above the visible board through `LevelConfig::spawn_coords`.
  * Gap: spawn cells do not match the spec exactly after translating to the project's 0-based coordinates; the current shape-offset spawn is one row too high for J/L/S/T/Z/O and two rows too high for I. The immediate one-row drop after Block Out is also not implemented.
* [x] Seven-bag randomizer.
  * Current: `PieceGenerator` shuffles one copy of each of the seven piece types and refills only when the bag is empty.
  * Gap: RNG is not seedable yet, which will matter for deterministic tests and multiplayer fairness.
* [ ] Next Queue 1..6, default ideally 6.
  * Current: the preview UI defaults to 6 pieces via `LevelConfig::preview_count`, using `PieceGenerator::preview`.
  * Gap: visible count is not configurable from 1 through 6, and there is no explicit Next Queue state initialized with `visibleNextCount + 1` pieces.

  * Current: Hold exists, is on by default, and `PieceController::used_hold` prevents a second Hold before the next piece.
  * Gap: Hold cannot be disabled, Hold swap does not check Block Out or immediate-drop rules, and the stored held `Piece` can keep its previous rotation instead of always respawning North Facing.
* [ ] Ghost Piece default on.
