# Lever #4 dossier — bitboard movegen (research, 2026-07-10)

**Headline: the Cold Clear 2 collision-maps change measures 12.4-13.5% (empirically rebuilt at its commit), not the estimated 25-30% — the lever is real but smaller than the ledger guessed; the rewrite risk stands. DEPRIORITIZED below simpler levers until the campaign demands it.**


## Cold Clear 2 (primary reference)

Cold Clear 2 (`src/movegen.rs`, `src/data.rs`) generates moves over a column-major bitboard: `Board.cols: [u64; 10]`, bit y of `cols[x]` = cell (x,y), 40 usable rows. The core trick is `CollisionMaps::new(board, piece)`: per call it builds `[[u64;10]; 4]` — for each rotation, a bitmap over piece-ORIGIN space where bit (x,y) is set iff the piece placed there collides. Construction: for each of the 4 rotated cell offsets (dx,dy), OR in column x+dx shifted by dy (dy<0 uses `!(!c << -dy)` to fill vacated low bits with 1s = floor; out-of-range x yields `!0` = solid wall). ~160 shift/OR ops total; afterwards EVERY obstruction test is one bit test (`boards[rot][x] & 1<<y`) instead of 4 per-cell lookups with bounds checks.

Search: Dijkstra-style best-first over `Placement = (piece, rotation, x, y, spin)` with cost = rows soft-dropped (shifts/rotations cost 0), `BinaryHeap<Intermediate>` + `AHashMap<Placement,u32>` distance map with stale-entry skip. Note the Ord lacks `.reverse()` so it's max-first — still exact (label-correcting via re-queue on improvement), just not true Dijkstra. Each expansion: sonic-drop (per-cell drop distance = one lzcnt: `(!col << (64-y)).leading_ones()`, min over 4 cells), record lock keyed by `canonical_form()` (folds S/Z/I 2-fold and O 4-fold rotation symmetry with x/y adjustment) keeping min soft-drops, then try shift ±1, rotate CW/CCW.

Fast mode (all column heights < 16): skips BFS-through-air entirely — enumerate 4 rotations × 10 x at y=19, hard-drop each via lzcnt, push locks directly (bypassing hashmaps), and seed the queue only with shift/rotate neighbors of the RESTING positions; `update_position` prunes any state fully above the stack (redundant — reachable at spawn height). This is the largest win vs naive BFS-from-spawn.

SRS: kick tables are const-derived at compile time from the guideline offset formulation — `kicks(piece,from,to)[i] = offsets(from)[i] − offsets(to)[i]`, 5 kicks, separate I/O offset tables, indexed by CC2's enum order N,W,S,E via `piece_lut!`/`rotation_lut!`. Each kick test is one collision-map bit test. T-spin classification happens inside `rotate()` on the first non-colliding kick and needs the RAW board (not the map): count occupied among 4 diagonal corners (OOB counts occupied — walls/floor give corners); <3 → None; else Full if both front corners (rotate_cell of (−1,1),(1,1)) filled OR kick index i==4 (TST/fin rule), else Mini. Spin survives only if the node is grounded: shift() resets spin, and the drop step keeps spin only when drop_dist==0 ("last action was rotation"). Spin is part of Placement identity, so same-location/different-spin are distinct search nodes. Only T gets spins (guideline, no all-spin).

Speedup: commit ed8b1932 (2022-06-29, "build valid-location lookup maps to avoid a loop for collision checking") introduced CollisionMaps. I rebuilt that commit and its parent and interleaved 3 timing rounds on the four benches/movegen.rs boards (Apple Silicon, release): empty 1637→1424 ns, tspin 1597→1399, dtd 1795→1566, terrible 18591→16081 ns per find_moves — a consistent 12.4–13.5%, identical move counts. The ~25–30% figure is plausible vs a truly naive per-cell BFS (the parent already had fast mode, const kick LUTs, canonical dedup, hashmap-free hard drops) or on x86 codegen; the mechanisms are: O(1) bit-test obstruction (amortizing piece shape into the precompute), lzcnt drop distances, air-BFS elision, and symmetry dedup.

Contrast, original cold-clear (`libtetris`): row-major `[u16; 40]` rows + cached column_heights, per-cell `occupied()`, Dijkstra over actual INPUT sequences (DAS timing: 1 tick/move, +1 double-tap, 2 ticks/row soft drop; modes ZeroG/ZeroGComplete/TwentyG/HardDropOnly), fast mode via a hardcoded `zero_g_starts` (position, input-list) LUT including double-rotate South entries — explicitly documented as possibly missing placements (ZeroGComplete disables it). Same SRS offset-derived kicks and identical 3-corner/i==4 T-spin rule in piece.rs.

**Risks:**
- Boundary fill in shifted-column map construction: dy<0 must fill vacated low bits with ones (`!(!c << -dy)`) for the floor, out-of-range x must read as all-solid (!0), and y<0 must remain an explicit check — an off-by-one here passes on low boards (fast mode masks it) and only misbehaves in high-stack Dijkstra mode
- Rotation enum ordering: CC2 uses N,W,S,E (not the guideline doc's N,E,S,W); every LUT (kicks, cells, collision maps) is generated in that order — porting published SRS kick tables into a different enum order silently swaps CW/CCW kick sets while still producing mostly-plausible moves
- Kick derivation sign: kick[i] = offset(from)[i] − offset(to)[i]; reversing the subtraction yields mirrored kicks that survive shallow tests (many kicks are symmetric) but break TST/fin/neo and wall kicks
- T-spin classification requires the raw per-cell board, not the collision map — corner tests use occupied() with out-of-bounds counting as filled (walls/floor legitimately supply corners); dropping the per-cell accessor when switching to mask-only collision breaks spin detection
- The Full-spin upgrade at kick index i==4 requires kicks tried in canonical SRS order with the index threaded through the collision loop; reordering or short-circuiting kicks demotes TSTs to Mini (wrong attack/B2B downstream)
- Spin must be part of search-node identity (Placement = location + spin) and must be reset on shift and on any nonzero drop ('last action = rotation'); deduplicating by location alone either loses executable T-spin placements or reports spins the path can't realize
- Canonical-form dedup must adjust coordinates when folding symmetric rotations (e.g., S South = North @ y−1, I West = East @ y+1, O all→North with x/y shifts) and must be applied only at lock-recording time — search states with coincident cells are still distinct (different spin/kick continuations)
- Fast-mode invariant is spawn-relative: heights<16 with spawn y=19 guarantees no top-row collision and full (rot,x) reachability above the stack; a different spawn row, buffer layout, or bigger kick range (e.g., TETR.IO SRS+ 180 kicks) invalidates the threshold and the above_stack prune silently drops legal placements
- CC2's BinaryHeap Ord lacks .reverse() (pops max soft-drops first): exactness is rescued only by the stale-value skip + re-queue-on-improvement; copying the pattern without the stale check, or copying CC1's or_insert lock dedup (which assumes min-first pop order) without min-ordering, gives wrong minimal-cost paths
- Guideline-only spins: CC2 classifies T-spins exclusively; for all-spin rulesets (TETR.IO all-mini) you need immobility or per-piece corner rules instead — the collision map makes immobility cheap (4 bit tests: up/down/left/right all obstructed) but that logic doesn't exist in either CC codebase to copy

## Ecosystem survey

Four distinct fast-movegen families exist beyond Cold Clear.

1) Row-bitmask + BFS pathfinding (MisaMino lineage). MisaMino (C++) stores the field as `unsigned long m_row[AI_POOL_MAX_H]` with `m_w_mask=(1<<w)-1`, and offsets the row pointer (`&m_row[gem_add_y]`, 20 rows) so pieces above the visible field use negative indices — a clean solution for spawn/buffer rows. Pieces are 4x4 per-rotation row bitmaps plus precomputed per-column heights (`GEM_COL_H[piece][spin][x]`). Kicks are hardcoded `wallkickdata` / `Iwallkickdata` indexed [state][direction][test][xy] — SRS-shaped but with y-down sign flips; spawn at (3,1). Movegen is `GenMoving` (reachable placements) + `FindPathMoving` (input reconstruction) + a max-heap best-first search; each Moving carries a `wallkick_spin` flag, and `isWallKickSpin` implements both 3-corner T-spin and an all-spin (L/R/top blocked) variant. Zetris is this engine ported/tuned as a PPT/PPT2/TETR.IO bot; its author's PerfectClearNET wraps knewjade's sfinder-cpp for PC play.

2) Whole-board packed u64 (knewjade / wirelyre). solution-finder packs the 10-wide field LSB-first, `x + y*10`, 6 rows per 64-bit long (Small/Middle/Large fields of 1/2/4 longs); `canPut` is a single AND against precomputed mino masks; line clear, slide, merge are all bit ops. sfinder-cpp is the C++ port. tetra-tools' srs-4l crate is the cleanest Rust reference: a `Copy` u64 board (bottom 40 bits = 4x10), piece bits computed as `PIECE_SHAPES[shape][orientation] << (row*10 + col)`, collision = AND, full SRS with 5-entry CW kick tables (CCW derived by mirroring/subtracting), plus bitwise unreachable-region culling (`has_isolated_cell`).

3) Column-major bitboards. The arXiv "bitboard Tetris AI" RL framework uses one 32-bit int per column (LSB=bottom), collision by shifting piece-column bits to the landing height and ANDing, O(log h) height via binary search, 53x over gym-tetris — but rotation is simplified rotate-in-place with NO kicks and drop-only placement. Useful for the column-major mechanics; useless as a rules reference.

4) Bit-parallel reachability ("waveform", HATETRIS record posts). Instead of BFS node-at-a-time, one u64 per fall depth encodes legality of every (x, rotation) simultaneously (4 rotation bits per x-offset); left/right = shift by 4, rotation = in-block permute + cross-block shift, collision = ANDs against masks precomputed for all 1023 line patterns over a 4-row slice. ~5us per full movegen, 10x over a cached HashMap. This is the highest-leverage idea for our beam-search bookkeeping wall, but it was built for HATETRIS's kickless rotation — SRS kicks and spin provenance must be re-derived.

Ecosystem: Hikari (SoRA_X7, GitLab, Unity C# Jobs/Burst) is the other serious PPT-era bot; ShakTrisLib (C++20 bitboard+BitPiece movegen) powers current Botris-community bots; the TBP spec pins the interop convention (SRS presumed, piece centers = SRS stationary points, spin field on moves). Canonical rule data: tetris.wiki SRS page and four.lol's kicks overview (which also documents TETR.IO's SRS+ 180 kicks — a divergence from guideline, and from our engine, that many community bots bake in). Most "tetris-ai" ML repos (gym wrappers, genetic bots, gotetris) use drop-only simplified rules — fine for speed comparisons, unusable for engine-exact contracts.

**Risks:**
- Kick-table coordinate conventions: guideline tables are y-up; most implementations (MisaMino included) are y-down with flipped signs, and some (tetra-tools) store only CW kicks and derive CCW by negation. A sign or ordering slip changes WHICH of the 5 kick tests fires first — same final cell, wrong kick index — which corrupts T-spin vs Mini classification (kick test 4 upgrades to full T-spin under guideline rules), not just reachability.
- Most fast-bitboard references implement simplified rules: rotate-in-place with no kicks and drop-from-above placement only (arXiv paper, gotetris, nearly all RL/tetris-ai repos). Porting their collision tricks wholesale silently drops floor kicks, TSTs, tucks, and slides — every placement class our engine-exact contract needs.
- Column-major height-shift collision (shift piece column bits by column height, AND) only models convex drop-from-above placements; SRS reachability requires BFS/bit-parallel flood over (x,y,rotation) including overhangs and tuck-unders, i.e. per-column AND at arbitrary y, not height-anchored.
- Piece-mask precomputation must encode true SRS rotation centers via the offset scheme: I and O move their bounding-box origin between rotation states. Naive 4x4-bitmap rotation places I/O wrong relative to kicks; MisaMino handles this with per-rotation bitmaps plus its own kick data, tetra-tools by baking offsets into per-orientation shapes plus adjusted kick tables — mixing conventions from two sources is a classic silent-corruption path.
- Spawn/buffer fidelity: guideline spawns flat-up in rows 21/22 above the visible field, and kicks/line clears can occur in buffer rows. Fixed-height packed boards (srs-4l's 4 rows, sfinder's 6-row longs) can't express this; MisaMino's offset-row-pointer (negative indices) is the pattern to copy. This mirrors our existing lock_clear buffer-zone-rows lesson.
- Spin provenance under bit-parallel movegen: waveform-style reachability computes position sets, losing per-path 'last move was rotation' and 'which kick' info that T-spin/Mini/all-spin classification requires. Deduping search nodes on (x,y,r) alone merges spin and non-spin arrivals at the same cell — the transposition table must key or maximize over spin state, or T-spin attack gets mispriced.
- 180 rotations: guideline SRS has no 180 kicks; TETR.IO's SRS+ and MisaMino/Zetris configs add them (four.lol documents the tables). Importing community kick tables or movegen fixtures that include 180 kicks generates moves our engine will reject — validate any borrowed table against tetris.wiki guideline SRS, per-piece, per-transition.
- MisaMino's kick data predates modern SRS documentation and its all-spin mode is nonstandard; treat its tables as 'SRS-like', verify offset-by-offset before reuse. Zetris's fidelity fixes live in the fork's diffs, not in any README — the only reliable check is differential testing of kick outcomes against our engine across all (piece, state, direction, occupancy) cases.

## Local integration constraints

CURRENT MOVEGEN (crates/tetr-core/src/ai/movegen.rs): FIFO BFS over ActivePiece states, generic over `Occupancy`. Start pose is normalized via `ActivePiece::at_pose` (spawn-fresh history — the start never classifies a spin). Expansion order per node is fixed: [Left, Right, Cw, Ccw, SoftDrop]; SoftDrop is sonic (loop try_move Down to floor; None if resting). Visited key = (PoseKey=(x,y,rot u8), spin_rank); emission dedup = (PoseKey, classification_key) with first (BFS-shortest) arrival winning per (pose, class). spin_rank (T only; others always 0): 0=shift/drop-final, 1=rotate-final kicks 1-4, 2=kick-5 rotate-final OR sticky used_kick_5_into_t_slot. classification_key = classify_t_spin at rest (0/1/2 = none/Mini/Full). movegen::rotate() mirrors api.rs's §7.5 override exactly: kick_number==5 && T && is_t_slot(landed pose) sets the sticky flag, which survives later moves (§12.4; move_to clears rotation records but never the sticky bit). Output is stable-sorted by (rotation u8, x, y, used_hold) — intra-pose duplicates (spin vs non-spin arrival, identical sort key) keep BFS emission order. This order is a load-bearing canonical tie-break: search/mod.rs best_root_plan takes the FIRST maximum in list order, shared by greedy/beam/best-first/policy planners and research minds via the single seam hold_placements(). Pathful/pathless duality: generate_with_hold (ply-1; paths rendered to engine inputs by plan.rs placement_to_inputs) vs generate_with_hold_pathless (interior plies, the W-times-per-beam-generation hot path) — pinned by test pathless_matches_pathful_modulo_paths (same placements, same order, empty paths). Gold contract: placement_classification_matches_the_engine_award replays paths through a real Engine and requires the awarded T-spin to equal the predicted one. Hold: swapped piece = hold.or(queue_front), used_hold tag + Hold path prefix; availability enforced upstream.

EXISTING BITBOARD (engine/bit_board.rs): column-major, cols:[u64;16] (MAX_WIDTH=16), bit y ⇔ (x,y) filled, ≤64 total rows; Copy. Primitives: Occupancy::blocked (walls/floor blocked, y≥64 free), occupied/set/clear, columns() -> &[u64], full_rows (AND-fold), clear_full_rows (iterative re-scan compaction), lock_piece -> LockOutcome, insert_garbage_lines, highest_y — all differential-tested against the engine Board. SearchState.board is already a BitBoard; movegen currently probes it cell-by-cell through collide_with. NOT present: per-(piece,rotation) column masks, shifted-collision caches, reachability planes — a rewrite builds these (piece bbox ≤4 cols × ≤4 rows → per-relative-column u8 masks shifted by y). SRS source of truth: pieces.rs try_rotate_with_kicks — I_KICKS/DEFAULT_KICKS (8 transition rows × 5 offsets, first (0,0)), 1-based kick numbers, O returns kick 0 no-op (movegen drops it, so O has one rotation plane); KICK_INDEX maps adjacent-orientation transitions.

FEASIBILITY: column-mask reachability CAN coexist with SRS + T history. Design: per (rotation, stratum) planes of per-column u64 reachable-sets; lateral spread = cross-column AND with free, iterated; sonic drop = per-column resting-row bit trick; rotations = 5 ordered shift+ANDNOT passes per transition (preserving first-fitting-kick priority so kick numbers stay correct); iterate planes to closure (upward kicks raise y, enabling new lateral rows). T needs 3 strata (ranks 0/1/2) — tractable, not a slow-path necessity — but exact enumeration parity is the hard part (see risks); the pragmatic split is bitboard for the pathless interior seam, keep the engine BFS for pathful ply-1 (paths must stay renderable), with the pathless==pathful test as the cross-implementation gate.

**Risks:**
- Enumeration-order parity: the canonical sort key (rotation,x,y,used_hold) excludes classification, so duplicate emissions at one pose (spin vs non-spin) are ordered by BFS-shortest emission via stable sort. A fixpoint/plane rewrite has no shortest-path notion; if intra-pose duplicate order flips, best_root_plan's first-max tie-break changes moves across every planner and silently invalidates golden baselines and the registry APP campaign numbers.
- Sonic-drop fidelity: Move::SoftDrop means fall-to-floor, not per-cell gravity. A naive bitboard fall-smear (stop at any height, spread laterally) over-generates placements that are unreachable under the real move alphabet and unrenderable by placement_to_inputs. Lateral spread is legal at any REACHED row (spawn row, resting rows and their hover continuations, kick-displaced rows) — restrict the plane transitions to exactly {shift, sonic-drop, kick-rotation} closure.
- SRS kick priority: try_rotate_with_kicks resolves the FIRST fitting of 5 ordered offsets and the 1-based kick number feeds spin_rank and the kick-5 rules. Bit-parallel rotation must be 5 ordered shift+ANDNOT passes (mask out already-resolved sources), never a union of kick offsets — a union mislabels kick numbers and corrupts T-spin classification.
- T strata and the rank-2 conflation: T needs ≥3 reachability strata (rank 0/1/2). Rank 2 currently conflates sticky (kick-5 INTO a slot; survives moves at rank 2) with transient kick-5-not-into-slot (decays to rank 0 on the next move) — they have different successors but share a visited key, so BFS first-arrival-wins shadows one. Exact parity means reproducing this conflation; splitting into 4 strata is more principled but CHANGES enumeration in edge cases — either way it must be a deliberate, differential-tested decision, not an accident.
- Kick-5 sticky probe timing: the sticky flag is decided by is_t_slot at the pose the rotation LANDS at, at rotation time (movegen::rotate mirrors api.rs), and survives all later non-rotation moves (§12.4). A rewrite that classifies only at rest misses spins where the kick-5 slot entry happens mid-path.
- Emission dedup is by classification-at-rest, not by stratum: rank-1 and rank-2 arrivals at the same pose can both classify Full and must dedup to one placement (first in BFS order). Deduping on (pose, rank) instead would emit extra duplicates and break no_duplicate_pose_classifications.
- Upward kicks (e.g. (0,2),(1,2), I's (1,2)) raise y and open new lateral rows — the plane fixpoint must iterate to closure across all rotation planes, not run a single down-sweep pass.
- Path tracking: the pathful ply-1 variant needs faithful button sequences; bitboard planes would need per-cell parent provenance to reconstruct them. Recommended integration: rewrite only generate_with_hold_pathless / hold_placements_pathless (the hot seam), keep the engine BFS for ply-1, and promote pathless_matches_pathful_modulo_paths into the cross-implementation differential gate over randomized boards.
- Geometry edge cases: start pose must be seeded with spawn-fresh history (rank 0, never classifies even if resting); O has exactly one rotation plane (kick-0 no-op is dropped); walls come from `width` not the u64 (blocked() treats y≥64 as free space above); buffer rows above visible height are real placement space (with_top_margin boards in tests).
